## Overview

This document describes the UI for the RFT (Reinforcement Fine-Tuning) product. The product helps users create fine-tuned models from their LLM gateway traces.

**Key Concept:** Our platform is an LLM Gateway. Users route their LLM traffic through us, generating traces. This product transforms those traces into fine-tuned models.

**Design Philosophy:** The main interface is a **visual canvas** showing the pipeline as connected nodes (similar to Figma/Stitch). Records are accessed via contextual overlays from the canvas.

# Part 2: Create Dataset (Single Screen)

## Screen: Create Dataset

**Route:** `/optimization/new`  
**Purpose:** Single screen to filter traces, configure dataset, and start pipeline.

This screen adapts based on trace availability:
- **Normal state:** User has traces, can filter and select
- **Empty state:** No traces in database (edge case for new users)
- **No results state:** Filters return zero matches

### Behavior

#### Filter Changes
- Immediately update match count
- Update trace list (virtualized for performance)
- Clear selection if filtered traces change
- Debounce search input (300ms)

#### Selection Changes
- Update "Selected: X traces" counter
- Update model breakdown chart
- Trigger AI detection on right panel (debounced 500ms)

#### AI Detection (Right Panel)
When traces are selected:
1. Analyze common system prompt patterns
2. Detect tool definitions
3. Identify capabilities (analysis, generation, etc.)
4. Auto-suggest dataset name (slugified)
5. Pre-fill objective based on detected capabilities

| State | Condition | Button |
|-------|-----------|--------|
| Disabled | No traces selected | Greyed out |
| Disabled | Form incomplete | Greyed out |
| Enabled | Traces selected + form complete | Active |

1. Create dataset record in database
2. Extract selected traces as records
3. Navigate to `/optimization/:id` (Canvas view)
4. Auto-start pipeline (Step 1 begins immediately)

### Validation Rules

Records are validated automatically. Invalid reasons include:

| Reason | Description |
|--------|-------------|
| No system message | Missing system prompt |
| No user message | No user turn in conversation |
| Empty assistant reply | Assistant response is empty |
| Tool call malformed | Invalid tool call JSON |
| Exceeds token limit | Conversation too long for training |
| Missing required field | Schema validation failed |

### Behavior

- **Auto-runs:** On initial load, after adding data, after editing records
- **Real-time:** Shows "Validating..." during checks
- **Non-blocking:** Pipeline continues with valid records only
- **Persistent:** Always visible, always up-to-date

# Part 7: State Management

## TypeScript Interfaces

```typescript
// Dataset
interface Dataset {
  id: string;
  name: string;
  objective: string;
  created_at: string;
  updated_at: string;
  stats: DatasetStats;
  pipeline: Pipeline;
}

interface DatasetStats {
  total_records: number;
  valid_records: number;
  invalid_records: number;
  generated_records: number;
  topics_count: number;
  balance_score: number | null;
  dry_run_status: 'pending' | 'go' | 'no_go' | null;
  dry_run_mean: number | null;
  training_runs: number;
}

// Pipeline
interface Pipeline {
  nodes: PipelineNode[];
  edges: PipelineEdge[];
  selected_node_id: number | null;
}

interface PipelineNode {
  id: number;
  name: string;
  status: NodeStatus;
  summary: string;
  position: { x: number; y: number };
  can_retrigger: boolean;
}

type NodeStatus = 'waiting' | 'running' | 'complete' | 'attention' | 'failed';

interface PipelineEdge {
  from: number;
  to: number;
  status: 'inactive' | 'active' | 'running';
}

// Records
interface DatasetRecord {
  id: string;
  dataset_id: string;
  messages: Message[];
  topic: string | null;
  valid: boolean;
  rejection_reason: string | null;
  dry_run_score: number | null;
  is_generated: boolean;
  source_record_id: string | null;
  created_at: string;
}

interface Message {
  role: 'system' | 'user' | 'assistant' | 'tool';
  content: string;
  tool_calls?: ToolCall[];
}

// Records Overlay State
interface RecordsOverlayState {
  is_open: boolean;
  context: RecordsContext | null;
  filters: RecordFilters;
  selected_ids: string[];
}

interface RecordsContext {
  source_step: number;
  label: string;          // "Invalid records (from Step 2)"
  preset_filter: RecordFilters;
}

interface RecordFilters {
  search: string;
  topic: string | null;
  valid: boolean | null;
  source: 'all' | 'traces' | 'generated';
}
```
